# PRD Quality Rubric — Per Section Checklist

## Section 1: Product Overview
- [ ] 2–4 sentences (not a single line, not a wall of text)
- [ ] States what the product does
- [ ] States who it's for
- [ ] States the core value delivered

**Failure mode:** "A loyalty app for coffee shops." — too thin, no value statement.

---

## Section 2: Problem Statement
- [ ] Describes a specific, real pain (not a generic market observation)
- [ ] Anchored in the target user's experience
- [ ] Explains why existing solutions fail
- [ ] Ends with a clear reason why this product exists

**Failure mode:** "People want better loyalty programs." — vague, no user context.

---

## Section 3: Target Users
- [ ] Primary user is specifically defined (not "small businesses" or "everyone")
- [ ] At least one explicit "out of scope" user group
- [ ] Platform or deployment context is noted if relevant

**Failure mode:** Target user is a broad category with no differentiating context.

---

## Section 4: User Personas
- [ ] At least 1 named persona (first and last name)
- [ ] Each persona has: role, goal, frustration
- [ ] Frustration is specific and traceable to the problem statement
- [ ] At least one persona has a direct quote

**Failure mode:** Personas without names or frustrations. Generic "user who wants to save time."

---

## Section 5: Jobs To Be Done
- [ ] 3–5 JTBDs
- [ ] Each follows: "When [situation], I want to [motivation], so I can [outcome]."
- [ ] JTBDs are solution-agnostic (not "use the app to track visits")

**Failure mode:** JTBDs are feature descriptions, not jobs.

---

## Section 6: Proposed Solution
- [ ] Describes the approach, not the technology
- [ ] Connects back to the problem statement
- [ ] Explains differentiation from existing alternatives
- [ ] Does not promise features not in the feature list

**Failure mode:** Reads like a tagline. No differentiation or problem connection.

---

## Section 7: Feature List
- [ ] 10+ candidate features total
- [ ] Features are grouped by category (Core / Engagement / Operational / Growth)
- [ ] Each feature has a one-sentence user-facing description
- [ ] No technical implementation details used as feature names

**Failure mode:** Features listed as bullet points with no descriptions.

---

## Section 8: MVP Scope
- [ ] MVP includes 5–8 features (not more)
- [ ] Each MVP feature has a rationale for inclusion
- [ ] Deferred features are listed with reason for deferral
- [ ] One "MVP success condition" statement is present

**Failure mode:** MVP includes 12+ features. No rationale for what's in or out.

---

## Section 9: Feature Prioritization
- [ ] Table format with Value, Effort, and Phase columns
- [ ] All MVP features appear in table
- [ ] Prioritization rationale paragraph explains the logic

**Failure mode:** Features listed without scores or no rationale.

---

## Section 10: User Stories
- [ ] At least 1 user story per MVP feature
- [ ] All stories follow: "As a [persona], I want to [action] so that [outcome]."
- [ ] Persona names reference actual personas from Section 4

**Failure mode:** Stories use "user" instead of persona names. Missing "so that" clause.

---

## Section 11: Success Metrics
- [ ] One clearly defined North Star metric with a numeric target
- [ ] Secondary metrics table with definitions, targets, and timeframes
- [ ] All targets include a number and a time window
- [ ] At least one anti-metric is stated

**Failure mode:** Metrics like "increase engagement" with no numbers or timeframes.

---

## Section 12: Risks and Assumptions
- [ ] 3–5 explicit assumptions listed
- [ ] Risks table with Likelihood, Impact, and Mitigation columns
- [ ] At least one open question listed
- [ ] Mitigation strategies are actionable (not "we'll monitor closely")

**Failure mode:** Generic risks like "competition" with no mitigation. Assumptions section is empty.

---

## Overall Pass Criteria

The PRD passes review when:
- All 12 sections are present
- All section-level checks pass (or pass after targeted fixes)
- The document reads clearly without product or engineering jargon
- A non-technical founder could pick it up and understand the full product vision
