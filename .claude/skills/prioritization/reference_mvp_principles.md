# MVP Principles and Decision Framework

## What an MVP Actually Is

An MVP is the smallest deployable product that:
1. Delivers real value to a real user
2. Can be tested with real users in the real world
3. Teaches you whether your core hypothesis is true

An MVP is NOT:
- A prototype or mockup
- An internal demo
- A feature-complete product with fewer features
- "Everything but the nice-to-haves"

---

## The MVP Hypothesis Test

Every MVP should be designed to test one core hypothesis:

> "If we build [X], then [target user] will [behavior/outcome] because [reason]."

Example:
> "If we build a QR-based digital stamp card, independent coffee shop owners will adopt it over paper punch cards because it runs automatically with no ongoing effort."

The MVP feature set should be the minimum needed to test this hypothesis.

---

## Common MVP Mistakes

**Adding "just one more thing"**
Each addition increases time to launch and reduces learning speed. Treat every addition as a trade-off.

**Confusing "core" with "complete"**
Core means: required for the value proposition. Complete means: all the features you imagined. These are different.

**Building for the 10% edge case**
Edge cases are post-MVP. The MVP validates the 90% use case.

**Including growth features in MVP**
Referral programs, sharing, integrations — these are post-MVP unless growth is the hypothesis.

**Building admin before product**
Operators often want dashboards and settings. Users need the core experience first.

---

## The Cut/Keep Decision Framework

When the operator wants to add something to MVP, ask:

1. **JTBD test**: Which job does this fulfill that isn't covered by what's already in MVP?
2. **Blocker test**: Will real users refuse to use the product without this feature?
3. **Learning test**: Does including this help or hinder our ability to test the core hypothesis?
4. **Swap test**: If we add this, what do we cut to keep scope manageable?

If the feature cannot pass at least 2 of these 4 tests, it goes to Post-MVP.

---

## Post-MVP Backlog Structure

Organize deferred features by phase:

**V1.1 (Fast follow — within 4 weeks of launch)**
Features that users will request immediately after using MVP

**V2 (Next quarter)**
Features that require validation from MVP usage data before building

**Future consideration**
Features that sound good but need market signal before committing
